Proposed PR sequence

  • PR-1 Delete targeting
  • PR-2 Harden API
  • PR-3 Client cleanup
  • PR-4 Decouple store

PR-1 · Remove targeting/H2 subsystem

low risk

Repo: persistence-service · pure deletion, verified self-contained.

  • Delete modules lottery-persistence + lottery-data-models
  • Delete LotteryTargeting[Summary]Controller, H2DatabaseConfig
  • Delete application-h2.yml, schema.sql, data.sql, cleanup cron
  • Drop JPA/H2 deps; remove default h2 profile

Kills: in-memory data loss, duplicate LotteryDraw, ~30 endpoints, unused cron.

PR-2 · Harden state-games API

medium risk

Repo: persistence-service · touches error/response contract.

  • @ControllerAdvice structured errors; correct 5xx vs 404 (stop returning 400 + null on server faults)
  • Fail-loud .ser reads: distinguish missing vs corrupt; log at ERROR
  • Path-traversal validation on state/game before building file paths
  • Configurable storage root (replace hardcoded "tmp/")
  • Dockerfile openjdk:8 → temurin:17 (currently cannot run the jar)
  • Drop duplicate POM blocks + unused resilience4j; resolve web-vs-webflux

PR-3 · Client cleanup

medium risk

Repos: lotto-options-core, lottooptionspro.

  • core: delete LotteryPersistenceClient + util/* targeting tooling + config
  • core: fix the double-slash bug /api/v1//state-games/
  • lottooptionspro: remove /lottery-targeting/configurations calls
Removing configurations cuts SmartNumberGenerationService's data source in lottooptionspro — see open question 1.

PR-4 · Decouple store (follow-on)

larger

Repo: persistence-service · read-path refactor, contract-affecting.

  • Separate lightweight catalog store from heavy draw history
  • Move durable format off Java serialization (JSON-on-disk / embedded DB)
  • Cache catalog; lazy-load draws only for POST /state/games
  • Publish OpenAPI; version the models jar off floating 0.0.1-SNAPSHOT

Planned separately after Tier-1 lands. Not part of the first push.

Its SmartNumberGenerationService reads lottery configs from that endpoint. If we delete targeting, that data source disappears. How should PR-3 handle it?

2. web vs webflux stack (PR-2)

Controllers return Flux but all I/O is blocking, and both web + webflux are on the classpath. I want to standardize — the constraint is not changing the JSON shape consumers already parse.
