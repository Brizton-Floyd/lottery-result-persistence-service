## 1. Spec — owner: java-tech-lead, Phase 0 gate

- **PRD:** Inline below — bug fix; no separate PRD document required.
- **Approval:** `APPROVED 2026-06-20`
- **Audit:** 7-point audit completed inline (see Section 1.1).
- **Architecture constraints & decisions:**
  - **Fix source chosen: Option 1 — read `drawPositionCount` directly from the deserialized `game` object.**
    The `game` object returned by `getStateData(request).getLotteryGame()` at line 206 of `LotteryDataService.java`
    is a fully-hydrated `LotteryGame` deserialized from a `.ser` file. Every `.ser` file was written by either
    `TexasLotteryHistoryProcessor.serializeDrawResults()` (line 134) or `LouisianaLotteryHistoryProcessor.serializeDrawResults()`
    (line 89), both of which call `lotteryGame.setDrawPositionCount(lotteryGame.getLotteryDraws().get(0).getDrawResults().size())`
    before serializing. Therefore `game.getDrawPositionCount()` is always populated in the `.ser` file. The fix is a
    single-line addition: `lotteryGame.setDrawPositionCount(game.getDrawPositionCount())`.
  - **Option 2 (DB config lookup) is rejected** for this fix. Rationale: `LotteryDataService` is already injected with
    no Spring Data repos; wiring `LotteryConfigurationRepository` solely to replicate a value already present in the
    `.ser` adds a JPA dependency to the catalog read path for no gain. Furthermore `data.sql` seeds only 4 of the 17+
    catalog games (lotto_texas, cash_five, powerball, mega_millions), so it cannot serve the remaining 13. Option 1 is
    authoritative at the storage layer and requires zero new dependencies.
  - **Null-guard required:** `listFilesForFolder` must guard `game.getDrawPositionCount()` for the theoretical case
    that a `.ser` file pre-dates the processor fix (i.e., `drawPositionCount` was not yet set when serialized). A
    safe `if (game.getDrawPositionCount() != null)` guard is mandatory; the field must never be set to null on the
    new object if the source is null (skip the set, leave `drawPositionCount` null rather than throw NPE).
  - **Scope — v2 path only (primary), plus secondary audit of sibling catalog paths:**
    - `getAllStateLotteryGamesV2()` → `listFilesForFolder()` — **PRIMARY fix target** (this is the path Core calls).
    - `getAllStateLotteryGames()` (v1 map path) calls the **same** `populate()` → `listFilesForFolder()` —
      identical omission. FR-020-B patches this in the same commit at zero marginal cost.
    - `fetchStateGames(stateName)` — returns only `fullName`, no metadata at all (name-only stub). Not in scope;
      no downstream consumer relies on it for drawPositionCount.
    - `getStateData(request)` — returns the raw deserialized `LotteryGame` including all fields. Already correct;
      no change needed.
    - `getAllStateLotteryGames(state)` — returns `List<String>` of game names only. Not applicable.
  - **`LotteryGame` model serialization confirmed:** `LotteryGame.drawPositionCount` is declared `private Integer`
    at line 19, the class is annotated `@Data` (Lombok generates getter/setter) and `@JsonInclude(NON_NULL)`.
    Jackson will serialize `drawPositionCount` once it is non-null. No model change required.
  - **`drawHistoryCount` is also omitted** from `listFilesForFolder`; however it is not consumed by Core and is out
    of scope for FR-020. Note for future cleanup.

### 1.1 Seven-Point Architectural Audit

| # | Area | Assessment |
|---|------|------------|
| 1 | **Problem** | `GET /api/v1/all/v2/state-games` returns `drawPositionCount=null` for every game. The field is baked into every `.ser` file by the aggregator processors but the catalog read path builds a new `LotteryGame` shell in `listFilesForFolder` and never copies the field. Downstream: Core `CatalogController` reads `getDrawPositionCount()` → null → 0; BFF relays 0; Angular UI defaults to 5 position toggles regardless of game type (Lotto Texas shows 5 instead of 6). |
| 2 | **Users** | Downstream services: lotto-options-core (`CatalogController`, FR-018), lotto-options-bff, and the Angular Range Analysis view. End persona: lottery player using the Range Analysis feature with wrong position toggles. |
| 3 | **Scope** | **IN:** Populate `drawPositionCount` in `listFilesForFolder()` (used by both v1 and v2 catalog paths) by copying the value from the deserialized `game` object; null-guard for legacy `.ser` files; a unit/integration test asserting the value flows through to the HTTP response. **OUT:** `drawHistoryCount` omission in catalog path (separate cleanup); `fetchStateGames` stub (name-only, no consumers); Angular UI changes; Core or BFF changes (those were already written for FR-018 assuming persistence would return the correct value). |
| 4 | **Targets** | Single file: `lottery-result-persistence-server/src/main/java/com/floyd/lottoptions/service/impl/LotteryDataService.java`, method `listFilesForFolder`. Test: `lottery-result-persistence-server/src/test/java/com/floyd/lottoptions/controller/PersistenceServiceControllerTest.java` (extend) or a new `LotteryDataServiceTest.java`. |
| 5 | **Constraints** | Must not introduce new Spring beans or JPA repository injections into `LotteryDataService`. Must not break existing `.ser` deserialization for any state. Must compile and pass `mvn test` with no new test failures. One-line fix to the production path; primary risk is the null-guard logic. |
| 6 | **Dependencies** | No new dependencies. `LotteryGame.getDrawPositionCount()` is already on the model classpath (`lottery-result-persistence-service-models`). `LotteryDataService` already holds a reference to the hydrated `game` object at line 206. |
| 7 | **NFRs** | No latency impact (field already loaded from `.ser`; just not copied). No security implications. Logging: no new log lines required for the happy path; consider a `log.warn` if `game.getDrawPositionCount()` is null at catalog time (indicates a stale `.ser` needing re-aggregation). |

### QC Findings (all PASS)

**1. Correctness — copy logic and null-guard (FR-020-A)**
`listFilesForFolder` at lines 222–228 reads `game.getDrawPositionCount()` from the fully-hydrated
`LotteryGame` deserialized at line 210, null-guards it with `if (game.getDrawPositionCount() != null)`,
sets the value on the new catalog shell, and falls to `log.warn(...)` on the null branch.
The `setLotteryDraws(null)` call at line 229 follows after the new block — correct placement;
the draw-list nulling is independent of the count copy and order is immaterial here.
No NPE path exists. No fabricated value on null source. Matches the architecture decision exactly.

**2. Scope coverage — v1 and v2 paths (FR-020-B)**
Both `getAllStateLotteryGames()` (v1 map path, line ~135) and `getAllStateLotteryGamesV2()` (v2
list path, line ~150) delegate through the same private `populate()` → `listFilesForFolder()`.
The single patch site covers both. Confirmed by traceability matrix and the v1 unit test.
`fetchStateGames` and `getAllStateLotteryGames(state)` correctly excluded (name-only stubs, no
drawPositionCount consumer). No paths were missed; no out-of-scope changes were made.

**3. Effective Java / code conventions**
- Logger: `private static final Logger log = LoggerFactory.getLogger(LotteryDataService.class)` —
  idiomatic SLF4J, correct placement at class level, static-final, self-typed (EJ Item 68: refer to
  objects by their interfaces; SLF4J Logger is the interface, LoggerFactory the factory — correct).
- No new raw types introduced; all collection declarations in the changed block use properly typed
  generics pre-existing in the file (`List<LotteryGame>`, `Map<String, List<LotteryGame>>`).
- No new Spring beans, no new JPA injections, no new class-level dependencies — additive one-method
  patch only. Zero scope creep.
- One trailing blank line after the logger declaration at line 27 (minor style nit — pre-existing
  style in the file; not a defect).

**4. Test quality (FR-020-C)**
- `LotteryDataServiceTest` — new file, 3 tests. All use `TestableWithCustomRoot`, a subclass that
  redirects the filesystem root and stubs `getStateData`. This is the correct seam: `getStateData`
  is the boundary between `listFilesForFolder` and the `.ser` filesystem read path.
- Happy-path v2 test asserts `catalogGame.getDrawPositionCount() == 6` on the returned catalog
  `LotteryGame` object — tests the observable output value, not that a setter was called.
  Also asserts `getLotteryDraws() == null`, confirming the shell remains lightweight.
- Happy-path v1 test asserts `drawPositionCount == 5` on the v1 map path — correctly exercises
  FR-020-B independently.
- Null-source test: `assertThatNoException` wraps the full call chain; inner loop asserts every
  catalog game's `drawPositionCount` is null — correctly verifies both no-NPE AND no-fabrication.
- `@TempDir` and `@BeforeEach` give each test a clean filesystem state; no shared mutable static
  fields; tests are independent and deterministic.
- One observation (not a defect): `TestableWithCustomRoot.listFilesForFolderInternal` is a
  hand-rolled mirror of the production `listFilesForFolder`. This means if the production method
  gains additional field copies in future, the test mirror will silently drift. The tradeoff is
  acceptable given `listFilesForFolder` is `private` (no direct override possible) and the test
  does drive the real FR-020-A copy logic through `buildCatalog()` → `listFilesForFolderInternal`.
  A future refactor that makes `listFilesForFolder` package-private or extracts it to a testable
  collaborator would remove this mirror. Out of scope for FR-020.

**5. Scope discipline**
Non-Texas `.ser` files (Colorado, Arkansas, Louisiana, New Mexico, California) correctly emit WARN
and leave `drawPositionCount` null. The implementation did NOT attempt to fabricate values for stale
files or patch the aggregator processors. Out-of-scope items (`drawHistoryCount` omission, v1 stub)
were noted in the work order and left untouched. Full scope discipline observed.
