# Changelog — 2026-06-20

## lottery-result-persistence-service

[BUGFIX] Populate drawPositionCount in v2 (and v1) state-games catalog — root-cause fix for Lotto Texas pick-6 bug (FR-020)

Root cause: `LotteryDataService.listFilesForFolder` built a new `LotteryGame` catalog shell but never copied
`drawPositionCount` from the fully-hydrated `.ser`-deserialized game object. Both the v1 (`/api/v1/all/state-games`)
and v2 (`/api/v1/all/v2/state-games`) catalog paths call the same `populate()` → `listFilesForFolder()` chain,
so both returned `drawPositionCount=null` for every game.

Fix: null-guarded copy of `game.getDrawPositionCount()` onto the catalog shell, with a WARN log when the source
is null (indicating a stale `.ser` file that pre-dates the aggregator processor fix).

Verification: unit tests 3/3 green (`mvn test` exit 0); live smoke — Lotto Texas=6, Cash Five=5 confirmed.
Non-Texas states with stale `.ser` files correctly emit WARN and return null (no NPE).

Completes the pick-count chain: this service (FR-020) → lotto-options-core FR-018 → lotto-options-ui FR-019.

[TEST] New LotteryDataServiceTest — 3 tests covering happy-path v2, happy-path v1, and null/stale-.ser case (FR-020)
